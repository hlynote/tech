# RTC 技术概要：核心架构与工程实践

RTC（Real-Time Communication）通过全球分布式接入、智能路由与端到端优化，实现<span style="color:#E53935;"><strong>高可靠、高并发、低时延的实时音视频与互动通信能力</strong></span>，支撑语聊房、连麦、在线会议、远程协作、互动直播等场景。

## 1. RTC 是什么

RTC 是一种以“实时”为核心目标的通信系统。与传统点播/直播相比，RTC 更关注端到端时延、抖动稳定性、丢包恢复速度和互动体验一致性。

关键目标通常包括：

- 低时延（毫秒级到亚秒级）
- 高可用（跨地域、跨网络稳定连接）
- 高并发（大规模房间/多路互动）
- 跨平台兼容（iOS/Android/Web/Desktop）

## 2. 核心技术栈

### 2.1 会话与传输

- 信令：负责入会、协商参数、角色变更、流发布订阅等控制平面
- 媒体传输：通常基于 RTP/RTCP，承载音视频媒体平面
- NAT 穿透：依赖 ICE + STUN/TURN 建立可达链路
- 组网策略：P2P、SFU（主流）、MCU（特定场景）

### 2.2 音视频处理

- 音频编解码：常见 Opus，兼顾语音质量与带宽效率
- 视频编解码：常见 H.264/H.265/VP8/VP9/AV1（按平台和场景选择）
- 前处理：降噪、回声消除（AEC）、自动增益（AGC）、静音检测（VAD）
- 后处理：混音、音量指示、字幕/SEI、录制与转码

### 2.3 弱网对抗与拥塞控制

- 丢包恢复：NACK、FEC、关键帧请求（PLI/FIR）
- 自适应码率：根据带宽估计和网络反馈动态调整码率/分辨率/帧率
- 抖动缓冲：平衡“流畅性”和“低时延”
- 多路下发策略：分层编码（Simulcast/SVC）与按订阅质量转发

### 2.4 安全机制

- 传输加密：DTLS-SRTP（媒体）、TLS（信令）
- 鉴权与鉴别：Token、签名、时效与权限控制
- 风险防护：防盗链、防重放、敏感日志脱敏与审计

## 3. 典型系统架构

可抽象为三层：

1. 终端层：采集、编码、渲染、设备管理、用户交互
2. RTC 服务层：信令、接入调度、媒体转发（SFU）、质量监控
3. 业务层：房间管理、用户关系、计费、审核、消息与互动玩法

工程上推荐通过“协议抽象 + 适配器模式”隔离厂商 SDK，降低后续替换成本。

## 4. iOS 侧落地要点

- 设备权限：麦克风、相机、相册/屏幕共享权限文案与流程
- 音频会话：`AVAudioSession` 类别、中断恢复、蓝牙/听筒切换
- 编解码与渲染：`VideoToolbox`、`Metal/OpenGLES` 路径兼容
- 屏幕共享：`ReplayKit` + RTC 推流链路
- 生命周期：前后台切换、来电打断、锁屏场景的状态一致性

## 5. 质量指标与监控建议

建议至少建立以下可观测指标：

- 端到端时延（上行、下行、首帧）
- 丢包率、抖动、往返时延（RTT）
- 卡顿率、音频中断率、重连成功率
- 加入房间成功率、发布/订阅成功率
- 崩溃率与关键错误码分布

同时建立“用户主观体验”与“客观 QoS 指标”的关联分析，便于快速定位问题。

## 6. 常见工程问题

- 弱网下音画不同步：需联动 jitter buffer 与音频主时钟策略
- 大房间上下行压力高：需启用分层编码与订阅降级策略
- 跨区域质量波动：接入点调度与中继路径优化是关键
- 中断恢复失败：需补齐 token 续期、重入房、重发流状态机

## 7. 实施建议

1. 先定义 QoS 目标（首帧、时延、卡顿、成功率）再选型
2. 统一抽象 SDK 能力面，避免业务层直接耦合厂商实现
3. 从 Day 1 建立观测面板和回放链路，缩短排障路径
4. 针对弱网、打断、跨区建立专项压测与演练机制
