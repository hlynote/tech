# PartyClub iOS 面试题（基于当前项目）

本文档基于当前项目的模块化架构（`BaseModules`、`FeatureModules`、`FeatureCommon`）和实际业务场景（房间、麦位、举报、游戏、特效策略）整理，适用于 iOS 工程师技术面试。

## 一、项目与架构理解（必问）

- 你们项目里 `BaseModules`、`FeatureModules`、`FeatureCommon` 的边界是怎么划分的？举一个实际模块说明“为什么放在这里”。
- `PCVoxing` 和 `PCMainFeature` 的职责差异是什么？如何避免它们互相依赖导致耦合？
- 如果让你新增一个 `PC` 前缀的新功能模块，你会怎么设计 Pod 依赖关系，避免循环依赖？
- `PCCore`、`PCShared`、`PCTheme`、`PCRouter` 在你们项目中的定位分别是什么？哪些代码绝对不应该放进这些基础模块？
- 在模块化工程中如何做“能力下沉”与“业务上浮”的平衡？请给一个你实际做过的重构案例。

## 二、业务场景设计（语聊房/麦位/举报/游戏）

- 语聊房麦位相关状态（上麦、静音、踢麦、闭麦）你会如何建模？如何防止状态错乱？
- `Report`（举报）这类流程如何保证可扩展（举报类型新增、文案新增、上报字段变化）？
- 在房间内做游戏（`GameService`）时，如何处理“游戏状态”和“房间状态”并发更新？
- 对于房间内“浮层/横幅策略”（如 `EffectFloatingBannerStrategy`），你会如何做优先级和冲突管理？
- 假设服务端消息乱序到达，你会怎么保证前端最终状态一致？

## 三、Swift 基础与进阶

- `struct` 和 `class` 在你们场景里怎么选？请结合 ViewModel 或 Service 举例。
- `final` 在项目里有什么收益？什么时候不该加 `final`？
- `weak self` 在闭包中总是要加吗？哪些场景可以不加？为什么？
- 解释一下 Swift 的值语义在状态管理中的好处和坑点。
- 在大型业务代码中你如何避免 Optional 地狱？常用策略有哪些？

## 四、MVVM / ViewModel 设计能力

- 你们 `PCBaseNavVMViewController<ViewModel>` 这种基类方案的优缺点是什么？
- ViewController 和 ViewModel 的边界怎么定？哪些逻辑放错位置会带来长期维护问题？
- ViewModel 输出如果同时驱动多个 UI 区块，如何设计输出协议/状态，避免“回调地狱”？
- 你如何处理 ViewModel 生命周期和页面生命周期错位（例如页面销毁但请求回调还在）？
- 你会如何设计一个可测试的 ViewModel？

## 五、网络层与服务层（Service）

- `PCVoxingService` 这类超大 Service（上千行）你会怎么拆分？按领域、按协议、按用例，还是按数据源？
- 网络请求失败重试策略怎么做（指数退避、幂等、错误码分类）？
- 如何统一处理业务错误码和展示层提示，避免到处写 `if code == xxx`？
- 你如何做接口兼容（旧字段/新字段并存）并降低线上 crash 风险？
- 房间页高频请求和实时推送共存时，如何避免重复刷新和 UI 抖动？

## 六、UI 工程化与性能（SnapKit / Theme）

- 你们为什么要求所有尺寸都走 `.widthScale`？它的边界场景是什么（iPad、横屏、动态字体）？
- 用 SnapKit 写约束时，你遇到过哪些性能或维护问题？怎么优化？
- `PCTheme` 的价值是什么？如果设计师改主题色，如何做到最小改动范围？
- 如何避免“硬编码颜色/间距/字体”回流到业务代码？
- 复杂列表页（多个 section、动态 cell）卡顿时你会如何排查？

## 七、并发与线程安全

- 你在项目里如何管理主线程 UI 更新与后台数据处理？
- 多个异步事件同时修改同一状态时，如何保证线程安全？
- Combine（若项目中使用）里你如何处理订阅释放和背压问题？
- 如果房间消息非常频繁，如何做事件节流/合并，避免主线程阻塞？

## 八、稳定性与线上问题排查

- 线上出现“偶现 UI 状态错乱”，你的排查路径是什么？
- 如何设计关键日志埋点，能快速定位用户在房间中的行为链路？
- 崩溃治理你会看哪些指标（崩溃率、影响用户数、版本分布、机型分布）？
- 你如何做灰度发布与回滚策略，降低功能上线风险？
- 遇到不可复现问题，你会如何构造最小复现环境？

## 九、测试与质量保障

- 你们当前业务里哪些逻辑最值得做单元测试？为什么？
- ViewModel 单测如何 mock Service？你会如何设计协议层？
- UI 自动化测试在语聊房场景下的难点是什么？
- 你如何定义一个需求“可提测”的质量门槛？
- 如果赶工期，测试范围该如何取舍？

## 十、可深入追问的高阶题

- 如何从“功能开发者”成长为“模块 owner”？你会建立哪些工程规范？
- 你会如何评估一次重构是否值得做？用什么指标衡量收益？
- 如果要把 `PCVoxing` 做成可复用能力（给多个 App 用），你会先改哪些点？
- 在多人协作下，如何防止模块边界被逐渐侵蚀？
- 请讲一个你做过的“复杂问题拆解”案例，重点说权衡与取舍。

## 使用建议（面试官）

- 初级候选人：重点看工程规范意识、基础功底、问题表达能力。
- 中级候选人：重点看模块边界、业务抽象、线上问题处理能力。
- 高级候选人：重点看架构演进、复杂系统设计、稳定性与团队协作推进能力。
